## The problem this document solves

Every team knows what they are building. Nobody has written down how the pieces connect. That means each team is making local decisions that affect everyone else вҖ” and nobody finds out until something breaks.

This document maps the full data flow from repo to human. It identifies the four handoff points where teams need to agree on a contract before building independently. Get the contracts right and every team can build in parallel. Get them wrong and v1 needs a rewrite.

## The four contracts

These are the handoff points where teams must agree before building. Each one is a question that needs an answer вҖ” not a suggestion.

---

### Contract 1 вҖ” AnalysisContext schema
**Between:** Team 1 (writes it) and Team 2/3 (reads it) and Team 4 (UI reads it)

This is the most important contract. Everything downstream depends on it. If this schema is wrong, all three teams have to redo work.

**What needs to be agreed:**

| Field | Required format | Why |
|---|---|---|
| `golden_file` | Repo-relative path string e.g. `src/lib/attempt-result.types.ts` | Adapters need it to cite sources. UI needs it to make a clickable link. Prose won't work for either. |
| `confidence` | Integer 1вҖ“3 per file or per signal | Adapters include it in output. UI renders it as a visual signal. If it's descriptive text it can't be rendered programmatically. |
| `gaps` | Explicit array e.g. `["no clear testing convention found"]` | Adapters surface what they couldn't find. UI shows gaps honestly. If absent from schema, gaps get papered over or hallucinated. |
| `file_scores` | Array of `{file, churn, centrality, age_months}` | FileSelector needs this to pick files per adapter. UI needs it to show architecture diagram. |
| `coupling_pairs` | Array of `{file_a, file_b, degree}` | RulesAdapter shows coupled files together. UI shows hidden dependencies. |

**Who decides:** Team 1 proposes, whole team ratifies. Must happen before any adapter is written.

---

### Contract 2 вҖ” rules.yaml schema
**Between:** Team 2/3 (writes it) and Team 4 (UI reads it) and future SkillAdapter (reads it)

The rules schema is already partially defined in DECIDED.md. But it needs two additions before it is UI-ready.

**What needs to be agreed:**

| Field | Required | Why |
|---|---|---|
| `id` | вң… already decided | Unique identifier per rule |
| `rule` | вң… already decided | The rule in technical terms |
| `why` | вң… already decided вҖ” must stay required | Reasoning. Non-negotiable. Rules without why get ignored. |
| `example` | вң… already decided | Code snippet showing right and wrong |
| `golden_file` | вң… already decided | File this rule was derived from |
| `confidence` | вҡ пёҸ needs adding | Integer 1вҖ“3. How many files this rule was observed in. UI renders as a signal. |
| `explain` | вҡ пёҸ needs adding | Plain English explanation for a non-technical reader. One paragraph. Generated by LLM at same time as rule вҖ” zero extra cost. Powers "explain this to me" in the UI. |

**Who decides:** Team 2/3 proposes, Team 4 confirms UI can consume it. Must happen before prompt templates are finalised.

### Contract 3 вҖ” summary.md format
**Between:** Team 2/3 (writes it) and Team 4 (UI reads it)

`summary.md` is the only output currently designed for humans. Two questions need answering.

**What needs to be agreed:**

**Question A вҖ” Does `summary.json` get generated alongside `summary.md`?**
The markdown is for humans to read. The JSON is for the UI to consume without parsing markdown. Generating both at adapter time costs almost nothing. Not generating JSON means the UI has to parse markdown later вҖ” fragile and unnecessary.

**Recommendation:** Yes, generate both from day one.

**Question B вҖ” What structure does the summary have?**
The UI needs to know what sections exist to render them correctly. A free-form markdown file is hard to render in a structured UI. A defined structure вҖ” codebase overview, key patterns, what to read first, known gotchas вҖ” is easy to render and easy to generate.

**Recommendation:** Define the summary structure in the prompt template before writing it.

## Before and after the UI вҖ” what changes for the prompt and summary.md

The `--level` flag (`--level beginner` / `--level intermediate`) is not yet decided for v1. What is already decided is that the template system supports it: `rules_python.md` and `rules_typescript.md` are already in `prompts/templates/`. Two more files вҖ” `summary_beginner.md` and `summary_intermediate.md` вҖ” follow the same pattern. No pipeline change. No additional LLM call.

**Before the UI,** level is a generation-time parameter only. One flag selects one template. The adapter runs once and writes `summary.md`. A beginner summary has more context, more explanation, less assumed knowledge. An intermediate summary skips the basics and goes straight to the unusual patterns. The output filename is the same. The level is invisible in the artifact.

**After the UI,** the level is still invisible вҖ” and that is the point. The UI does not know or care which template was used. It renders `summary.md` by section. This only works if both templates produce the same section structure: identical headings, different depth. If the headings drift between templates, the UI needs two rendering paths. That is avoidable, and it must be avoided.

**The decision that must happen before any template is written:** lock the section structure. Not the content вҖ” the headings. What sections does `summary.md` always have, regardless of level? Once that is agreed, both templates are constrained by it, and the UI is constrained by nothing.


**Who decides:** Team 2/3 proposes structure, Team 4 confirms it's UI-consumable.

---

### Contract 4 вҖ” Experience layer interface
**Between:** Team 4 (builds it) and everyone (whose output it reads)

The experience layer reads from `analysis_context.json`, `rules.yaml`, and `summary.md/json`. It produces a local web UI.

**What needs to be agreed:**

- **Who owns it?** Right now nobody does. It needs an owner before v1 ships.
- **When does it ship?** v1 as a `--ui` flag that opens a browser, or v2? The flag should be reserved in v1 CLI regardless.
- **Where does it live?** Local only (runs on `localhost:3000`) or committed to the repo?
- **What does it read?** Exactly which fields from which files. Contracts 1-3 determine this.

**Technical reality:** Python's built-in `http.server`. Plain HTML + CSS + JS. Reads JSON files directly. No Node, no React, no build step. This is not a large engineering lift вҖ” it is a rendering layer on top of data that already exists.

**Who decides:** Whole team. This conversation needs to happen before v1 scope is locked.

### Whole team
1. Agree the four contracts before anyone builds independently

## The one thing that kills this project

Building in parallel without agreeing the contracts first.

If Team 1 locks the AnalysisContext schema without Team 2/3 and Team 4 input, adapters will need fields that aren't there. If Team 2/3 locks the rules schema without Team 4 input, the UI will have to parse markdown it shouldn't need to parse. If nobody owns the experience layer, it doesn't get built.

The contracts are small. The conversations are short. The cost of having them now is an hour. The cost of not having them is a rewrite.
