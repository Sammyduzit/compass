## The one-line version

Every competitor we researched produces output that developers don't actually use. Compass can be different — but only if four small decisions get made before the architecture is locked.

---

## Why this matters right now

We have done the research. We have built the mockup. We know what the experience layer needs to look like.

But the experience layer cannot be bolted on later without significant rework. The decisions being made this week — AnalysisContext schema, rules schema, CLI interface — either leave the door open or close it.

This is not a design conversation. These are four engineering decisions that take minutes to make and weeks to undo.

---

## Decision 1 — AnalysisContext schema (Team 1)

**The ask:** Three fields need to be UI-friendly from day one.

- `golden_file` — stored as a repo-relative path, not a prose string. A UI needs to turn this into a clickable link. If it's buried in text, a parser has to extract it later.
- `confidence` — stored as a number (1-3), not described in prose. The UI renders it as a signal. If it's prose, someone has to parse it later.
- `gaps` — an explicit array field for things Compass couldn't find. "We didn't find a clear convention for X" is better than a hallucinated rule. If gaps aren't a field, the UI can't surface them.

**Cost of doing it now:** 10 minutes of schema design.
**Cost of doing it later:** Rewrite the schema, re-run all adapters, update all consumers.

---

## Decision 2 — Rules schema (Team 2/3)

**The ask:** Add one field to each rule.

```yaml
- id: err-01
  rule: "Never use try/catch..."
  why: "Exceptions make failure paths invisible..."
  example: "..."
  explain: "In plain English: instead of your code crashing silently, this forces every failure to be visible. Think of it like a receipt for every error — you always know what went wrong and where."
```

The `explain` field is written for someone who doesn't know the jargon. It's generated by the LLM at the same time as the rule — zero extra cost. It's what powers the "explain this to me" interaction in the UI.

**Cost of doing it now:** One extra sentence in the prompt template.
**Cost of doing it later:** Re-run every adapter, update the schema, retrofit the UI.

---

## Decision 3 — CLI interface (whole team)

**The ask:** Reserve `--ui` as a recognised flag, even if it does nothing in v1.

```bash
compass /repo --adapters rules,summary --ui
# v1: runs pipeline, outputs files, flag is passive
# v2: opens browser to localhost:3000
```

If the flag isn't in the CLI design now, it gets deprioritised indefinitely. Reserving it costs nothing and signals that the experience layer is in scope.

**Cost of doing it now:** One line in the CLI argument parser.
**Cost of doing it later:** Nothing technically — but it never happens.

---

## Decision 4 — Experience layer ownership (whole team)

**The ask:** Someone needs to own this. Right now nobody does.

The pipeline has owners. The prompts have owners. The experience layer has a mockup and a brief but no assigned owner. That means it doesn't get built.

This does not have to be a large engineering lift. Python's built-in `http.server`. Plain HTML, CSS, JavaScript. Reads directly from `analysis_context.json` and `rules.yaml`. No Node, no React, no build step.

But it needs an owner before v1 ships — not after.

## The competitive context in one paragraph

We researched every tool in this space. Swimm, Greptile, Mapstr, Mintlify, Copilot. None of them extract team coding conventions automatically. None of them cite sources at the rule level. None of them are designed for how it feels to be new. Compass has an uncontested position in the market — but only if the output is something a new developer would actually use. A YAML file in a folder is not that. The experience layer is what makes it that.

---

## The design brief in one sentence

> Build something a new developer would screenshot and send to their team chat on their first day.

That is the bar. The pipeline is in service of it.
