# Compass — Codebase Summary

> Generated by Compass · 2026-04-20

---

## What this codebase does

Compass is a CLI tool that scans an unfamiliar codebase and produces two
artifacts: `rules.yaml` (the team's coding conventions, extracted automatically)
and `summary.md` (an onboarding overview). It works in two phases — first
collecting data from the repo using static analysis and git history, then calling
an LLM to synthesise that data into structured output. The pipeline is designed
to run once and persist its results, so the expensive collection phase does not
need to repeat every time you want to regenerate the output.

---

## Where to start reading

1. **`src/compass/domain/analysis_context.py`** — the data structure that connects
   the two pipeline phases. Understanding what it stores tells you how the whole
   system is shaped.

2. **`src/compass/adapters/base.py`** — the base class every adapter extends.
   Read this to understand the contract all output generation follows: declare
   your inputs, define your schema, implement `run()`.

3. **`src/compass/collectors/base.py`** — the base class every collector extends.
   Read this alongside `adapters/base.py` to understand the phase boundary —
   what is allowed in Phase 1 and what is not.

4. **`src/compass/runner.py`** — the pipeline orchestrator. This is where Phase 1
   and Phase 2 are sequenced. Read this once you understand the two base classes
   and you will see how everything fits together.

## What's stable

The domain and base class files are the load-bearing foundation of the codebase —
low churn, high centrality, and rarely touched:

- **`src/compass/domain/`** — data structures only, no logic. Stable by design.
- **`src/compass/collectors/base.py`** — the collector contract. Has not changed
  since the phase boundary was locked.
- **`src/compass/adapters/base.py`** — the adapter contract. Equally stable.
- **`src/compass/storage/`** — filesystem persistence. Low churn, owns all file
  writes for the pipeline.

These are safe to read, safe to copy patterns from, and unlikely to change while
you are getting oriented.

---

## What's actively changing

- **`src/compass/collectors/file_selector.py`** — per-adapter file selection logic.
  The `apply_coverage()` post-pass is still being tuned. Do not base assumptions
  on its current behaviour.

- **`src/compass/providers/`** — provider wrappers are in active development as
  Codex support is added alongside Claude. The interface is stable but
  implementations are changing.

Do not copy patterns from these files until they settle.

## How the pieces connect

**Collectors cluster** — `src/compass/collectors/`
All collectors share the `base.py` contract and produce sections of
`AnalysisContext`. `file_selector.py` and `repomix.py` are tightly coupled —
they always change together because FileSelector controls which files Repomix
processes. Touching one almost always means touching the other.

**Adapters cluster** — `src/compass/adapters/`
`rules.py` and `reconciliation.py` are coupled — RulesAdapter makes two LLM
calls and the reconciliation pass is the second one. `summary.py` is independent.
Both adapters depend on `base.py` but not on each other.

**Domain cluster** — `src/compass/domain/`
Standalone. No dependencies on collectors, adapters, or providers. Everything
else depends on domain — domain depends on nothing. If you are not sure where
a data structure belongs, it goes here.

**Providers cluster** — `src/compass/providers/`
Subprocess wrappers only. Depended on by adapters; depends on nothing inside
Compass. Adding a new provider means adding one file here and nowhere else.
